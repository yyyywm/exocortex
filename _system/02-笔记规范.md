# 笔记规范

## 1. Frontmatter 规范（所有笔记必须遵守）

### 必填字段

```yaml
---
title: "string"              # 笔记标题，与文件名一致（允许中文）
type: "atom"                 # 来自 01-架构规范.md 的 type 枚举
domain: "control"            # 来自 01-架构规范.md 的 domain 枚举
status: "seedling"           # seedling | budding | evergreen
draft: true                  # true（不发布）| false（发布）
date: "2026-06-06"           # 创建日期，YYYY-MM-DD 格式
---
```

### 选填字段（按 type 使用）

```yaml
mastery: 0.5                 # 0.0~1.0，仅 atom/compound 类型使用
difficulty: "intermediate"   # beginner | intermediate | advanced，仅 atom 使用
prerequisites:               # 前置概念列表，仅 atom/compound 使用
  - "前置概念A"
  - "前置概念B"
derived_concepts:            # 衍生概念列表，仅 atom/compound 使用
  - "衍生概念A"
code_refs:                   # 关联代码文件路径（相对于仓库根目录）
  - "sources/ardupilot/libraries/AP_Scheduler/AP_Scheduler.cpp"
tags:                        # 标签，自由但建议复用已有标签
  - "ardupilot"
  - "调度"
updated: "2026-06-06"        # 更新日期，YYYY-MM-DD
description: "string"         # 摘要，用于博客 SEO 和搜索引擎展示
sources:                     # 参考来源列表
  - "论文标题或 URL"
  - "书籍章节"
---
```

### 字段约束

| 字段 | 类型 | 约束 |
|------|------|------|
| title | string | 长度 1-100，不允许空 |
| type | enum | 必须在 type 枚举表中 |
| domain | enum | 必须在 domain 枚举表中 |
| status | enum | seedling / budding / evergreen |
| draft | boolean | true / false |
| date | string | YYYY-MM-DD 格式 |
| mastery | number | 0.0 ~ 1.0，保留一位小数 |
| difficulty | enum | beginner / intermediate / advanced |

## 2. 原子化标准（Atom 判定规则）

一条笔记是合格的 atom，**当且仅当**满足以下全部条件：

1. **单一性**：只讲**一个**概念、机制或原理
2. **独立性**：可以脱离其他笔记被理解（允许有 prerequisites，但正文必须自包含）
3. **完整性**：至少包含"直觉解释"和"形式化定义"之一
4. **适度性**：长度在 200~2000 字之间（纯代码实现类除外）
5. **可连接性**：至少能链接到 1 个前置概念和 1 个衍生概念（或明确标注待连接）

## 3. 内容模块标准

### 原子概念（atom）必备模块

```markdown
# {title}

## 直觉（Intuition）
用一句话或一个类比讲明白这个概念是什么。如果讲不明白，说明你还没真懂。

## 形式化定义（Definition）
数学定义、严格描述或精确的技术定义。

## 关键代码（Code）
```cpp
// From: sources/.../file.cpp:行号
```

## 关系图谱（Connections）
- 上游（依赖）：[[...]]
- 下游（被依赖）：[[...]]
- 相似/对比：[[...]]

## 我的理解 v1.0
记录当前的理解程度、局限性和待澄清问题。
```

### 代码解读（source）必备模块

```markdown
# {模块名} 源码解读

## 文件职责
一句话说明这个文件是干什么的。

## 关键函数
| 函数名 | 行号 | 职责 |
|--------|------|------|
| funcA() | L105 | ... |

## 核心代码片段
```cpp
// From: sources/.../file.cpp:105
```

## 调用关系
谁调用了它 → 它调用了谁

## 与知识库的关联
- 涉及概念：[[...]]
- 可对比实现：[[...]]
```

## 4. 链接规范

### 内部链接
- 使用 `[[笔记名]]` 建立笔记之间的语义关联
- 优先链接到原子概念（`10-Atoms/` 下的笔记）
- 复合系统笔记应大量引用原子概念

### 代码链接
- 使用 `[[sources/ardupilot/...]]` 直接链接到源码文件
- 行号引用使用 Markdown 标准链接：`[函数名](sources/.../file.cpp#L105-L130)`

### 外部链接
- 使用标准 Markdown `[文本](URL)`
- 论文链接优先使用 DOI 或 arXiv

## 5. 代码引用规范

嵌入代码必须同时满足：
1. **标注来源**：`// From: 相对路径:行号`
2. **最小化**：只嵌入核心逻辑，不要整文件复制
3. **可运行**：如果是 implementation 类型，代码必须能独立运行或通过简单修改运行

示例：
```cpp
// From: sources/ardupilot/libraries/AP_Scheduler/AP_Scheduler.cpp:105
void AP_Scheduler::tick(void)
{
    const uint32_t now = AP_HAL::millis();
    for (uint8_t i = 0; i < _num_tasks; i++) {
        if (_tasks[i].tick() == 0) {
            continue;
        }
    }
}
```

## 6. 理解版本标记

在笔记末尾固定使用以下格式：

```markdown
## 我的理解 v1.0

> 当前理解：...
> 已验证：...
> 待澄清：...
> 更新日期：2026-06-06
```

### 版本号规则
- **主版本升级（v1 → v2）**：认知框架发生质变，或发现之前理解有根本性错误
- **次版本升级（v1.0 → v1.1）**：补充细节、修正小错误、增加例子
- **补丁升级（v1.0.0 → v1.0.1）**：措辞优化、格式调整，不改变理解本身

## 7. 标签规范

标签使用小写，优先复用已有标签：

| 标签 | 用途 |
|------|------|
| `ardupilot` | ArduPilot 相关内容 |
| `control` | 控制理论 |
| `estimation` | 状态估计 |
| `embedded` | 嵌入式系统 |
| `rtos` | 实时操作系统 |
| `algorithm` | 通用算法 |
| `math` | 数学 |
| `draft` | 草稿状态（与 frontmatter draft 配合） |
| `evergreen` | 常青笔记 |

新增标签时应在 `90-MOCs/` 下创建标签索引页或在笔记中自然使用，不强制维护全局标签列表。
