# AI 操作规范

## 1. 角色定义

Claude Code 在本系统中的角色是**规范执行器（Specification Executor）**。

你不是自由创作者，而是按照本规范文档中的标准作业程序（SOP）执行操作。你的所有行为必须可追溯、可验证、符合 `_system/` 下的规范。

## 2. 操作边界

### ✅ 你可以做的
- 读取和创建 `content/` 下的笔记文件（`.md`）
- 读取 `sources/` 下的代码文件（**只读**，不修改）
- 读取所有 `_system/`、`templates/`、`.github/workflows/` 下的规范文档和配置文件
- 执行 Git 操作（`add`, `commit`, `push`, `status`, `log`）
- 执行 Quartz 构建命令（`npx quartz build` 等）
- 读取 `package.json`、`quartz.config.ts` 等配置文件

### ❌ 你不可以做的
- **修改 `_system/` 下的规范文档**（除非用户明确说"更新规范"）
- **删除已有笔记**（除非用户明确说"删除"）
- **修改 `sources/` 下的源码**（只读权限）
- **修改 Quartz 核心代码**（`quartz/` 目录）
- **修改 `.github/workflows/` 工作流文件**（除非用户明确要求）
- **修改 `.gitignore`**（除非用户明确要求）

## 3. 标准作业程序（SOP）

### SOP-01：从碎片创建原子笔记

**触发条件**：用户提供一段碎片内容或主题
**输入**：碎片文本 或 主题字符串
**输出**：一条符合规范的原子笔记文件路径

**执行步骤**：

1. **判断 domain**：根据主题内容，从 `01-架构规范.md` 的 domain 枚举表中选择最合适的 domain
2. **确定文件名**：将主题转换为 kebab-case，如 `"PID 控制器"` → `PID-控制器.md`
3. **确定文件路径**：`content/10-Atoms/{domain}/{文件名}`
4. **读取模板**：从 `templates/原子概念.md` 复制模板内容
5. **填写 frontmatter**：
   - `title`: 主题名（中文）
   - `type`: `atom`
   - `domain`: 步骤 1 确定的 domain
   - `status`: `seedling`
   - `draft`: `true`
   - `date`: 今天的日期（YYYY-MM-DD）
   - `mastery`: `0.1`
   - `prerequisites`: `[]`
   - `derived_concepts`: `[]`
6. **填充内容**：在"直觉"部分写入用户的碎片内容，或基于主题生成初始理解
7. **保存文件**
8. **Git 提交**：
   ```bash
   git add {文件路径}
   git commit -m "atom: {title}"
   ```
9. **返回**：文件完整路径

**验证标准**：
- [ ] 文件存在于 `content/10-Atoms/{domain}/`
- [ ] frontmatter 包含所有必填字段且格式正确
- [ ] 文件名符合 kebab-case
- [ ] Git commit 成功

### SOP-02：从源码生成代码解读笔记

**触发条件**：用户指定一个代码文件或函数
**输入**：代码文件路径（相对于仓库根目录，通常以 `sources/` 开头）
**输出**：一条 `source` 类型的代码解读笔记

**执行步骤**：

1. **读取代码文件**：验证文件存在且可读
2. **分析代码**：
   - 文件整体职责（一句话）
   - 关键函数列表（函数名、起始行号、职责）
   - 主要依赖的头文件/模块
3. **确定 domain**：通常为 `ardupilot` 或 `embedded`
4. **确定模块名**：从文件路径提取，如 `AP_Scheduler`
5. **创建笔记文件**：`content/50-Sources/Codebases/ArduPilot/{模块名}-解读.md`
6. **填写 frontmatter**：
   - `title`: `"{模块名} 源码解读"`
   - `type`: `source`
   - `domain`: 步骤 3 确定的 domain
   - `status`: `seedling`
   - `draft`: `true`
   - `date`: 今天
   - `code_refs`: `["{输入的代码路径}"]`
7. **填充内容**：按 `02-笔记规范.md` 中"代码解读必备模块"的结构填充
8. **尝试建立连接**：在"与知识库的关联"部分，基于代码内容建议可能的 `[[ ]]` 链接
9. **保存并提交**：
   ```bash
   git add {文件路径}
   git commit -m "source: {模块名} 代码解读"
   ```
10. **返回**：文件路径 + 关联建议

### SOP-03：建立笔记连接

**触发条件**：用户要求连接两条或多条笔记，或 AI 自动发现应连接的关系
**输入**：笔记文件路径列表（2 个或以上）
**输出**：更新后的笔记文件

**执行步骤**：

1. **读取所有指定笔记**
2. **分析关系**：确定笔记之间的逻辑关系（上游/下游、相似、对比、依赖）
3. **更新正文**：在每条笔记的"关系图谱"部分添加 `[[其他笔记名]]` 链接
4. **更新 frontmatter**（仅限 atom/compound 类型）：
   - 上游笔记加入 `prerequisites`
   - 下游笔记加入 `derived_concepts`
5. **提升成熟度**：如果连接数 >= 2 且 status 为 `seedling`，提升为 `budding`
6. **保存所有文件**
7. **Git 提交**：
   ```bash
   git add content/
   git commit -m "chore: 连接 {笔记A} 与 {笔记B}"
   ```
8. **返回**：更新的文件列表

### SOP-04：发布博客文章

**触发条件**：用户要求发布一篇博客草稿
**输入**：草稿文件路径（`content/80-Blog/drafts/` 下的 `.md` 文件）
**输出**：发布后的文件路径 + 网站 URL

**执行步骤**：

1. **读取草稿**
2. **质量检查**：
   - [ ] frontmatter 包含 `title`, `description`, `tags`, `date`
   - [ ] 正文长度 >= 500 字
   - [ ] 无明显的 `TODO`、`FIXME`、占位符
   - [ ] 所有 `[[ ]]` 链接指向的笔记存在（或已转换为标准链接）
3. **移动文件**：从 `content/80-Blog/drafts/{文件名}` 移动到 `content/80-Blog/posts/{文件名}`
4. **更新 frontmatter**：
   - `draft`: `false`
   - `date`: 今天的日期
5. **本地构建验证**（可选，如环境就绪）：
   ```bash
   npx quartz build
   ```
6. **Git 提交并推送**：
   ```bash
   git add content/80-Blog/
   git commit -m "blog: 发布 {title}"
   git push
   ```
7. **返回**：
   - 发布文件路径：`content/80-Blog/posts/{文件名}`
   - 预期 URL：`https://yourname.github.io/digital-garden/80-Blog/posts/{slug}`

### SOP-05：知识缺口分析

**触发条件**：用户指定一个学习目标或概念
**输入**：目标概念名或笔记文件路径
**输出**：缺口分析报告（Markdown 文件）

**执行步骤**：

1. **读取目标笔记**：获取其 `prerequisites` 列表
2. **递归检查**：对每个前置概念：
   - 是否存在对应笔记？（搜索 `content/10-Atoms/` 和 `content/20-Compounds/`）
   - 如果存在，检查 `mastery` 是否 >= 0.5
3. **生成分类**：
   - 🔴 **缺失**：前置概念无对应笔记
   - 🟡 **薄弱**：有笔记但 `mastery` < 0.5
   - 🟢 **已掌握**：有笔记且 `mastery` >= 0.5
4. **生成学习路径**：按依赖顺序排列建议学习的主题
5. **写入报告**：`content/90-MOCs/分析报告/{目标}-缺口分析-{日期}.md`
6. **Git 提交**：
   ```bash
   git add content/90-MOCs/分析报告/
   git commit -m "moc: {目标} 知识缺口分析"
   ```
7. **返回**：报告文件路径 + 摘要

---

### SOP-06：从 Inbox 批量整理

**触发条件**：用户要求整理 `00-Inbox/` 中的碎片
**输入**：无（自动读取 `00-Inbox/` 下所有文件）
**输出**：整理后的原子笔记 + 清空/归档 Inbox

**执行步骤**：

1. **读取 Inbox**：列出 `content/00-Inbox/` 下所有 `.md` 文件
2. **逐条分析**：对每条碎片：
   - 判断属于哪个 domain
   - 判断是否值得转为原子笔记（有价值 vs 垃圾）
   - 判断是否有现有笔记可以合并
3. **执行 SOP-01**：为有价值的碎片创建原子笔记
4. **处理剩余**：
   - 无价值的碎片：删除
   - 已处理的碎片：移动到 `content/00-Inbox/_archived/`（如存在）或删除
5. **Git 提交**：
   ```bash
   git add content/
   git commit -m "chore: 整理 Inbox，生成 {N} 条原子笔记"
   ```
6. **返回**：生成的笔记列表 + 删除的碎片列表

## 4. 决策树（未指定 SOP 时的默认逻辑）

当用户没有明确说"执行 SOP-XX"时，按以下规则判断：

```
用户输入是...
├── "记一下..." / "这个想法..." / 一段碎片内容 ──► 执行 SOP-01
├── "分析 XXX 代码" / "解读这个文件..." ────────► 执行 SOP-02
├── "连接 A 和 B" / "这两篇什么关系" ──────────► 执行 SOP-03
├── "发布 XXX" / "这篇可以发了" ──────────────► 执行 SOP-04
├── "我想学 XXX" / "搞懂 XXX 需要什么" ────────► 执行 SOP-05
├── "整理一下 Inbox" / "处理碎片" ────────────► 执行 SOP-06
└── 开放式技术问题 ────────────────────────────► 先搜索相关笔记，再基于笔记回答
```

## 5. Git 提交信息规范

所有 Git 提交必须按以下格式：

```
<type>: <描述>

[可选的详细说明]
```

### Type 枚举

| Type | 用途 | 示例 |
|------|------|------|
| `atom` | 新建或更新原子概念 | `atom: PID 控制器` |
| `compound` | 新建或更新复合系统 | `compound: ArduPilot 调度系统` |
| `source` | 新建或更新来源解读 | `source: AP_Scheduler 代码解读` |
| `derivation` | 新建或更新推导 | `derivation: 梯度下降收敛性证明` |
| `blog` | 发布博客文章 | `blog: 从零理解 ArduPilot 调度器` |
| `moc` | 更新内容地图 | `moc: 添加 ArduPilot 学习路径` |
| `insight` | 新建洞察 | `insight: 关于 EKF 初始化的陷阱` |
| `chore` | 杂项（连接笔记、格式调整、模板更新） | `chore: 连接 PID 与卡尔曼滤波` |
| `system` | 更新系统规范（**需用户明确授权**） | `system: 更新 AI 操作规范` |

## 6. 错误处理与边界情况

执行 SOP 时遇到以下情况，**立即停止并报告用户**：

| 情况 | 处理方式 |
|------|----------|
| 模板文件缺失 | 报告：`templates/xxx.md 不存在，无法执行 SOP-XX` |
| 目标目录不存在 | 报告：`目录 content/xxx/ 不存在，架构可能被修改` |
| 用户要求超出操作边界 | 报告：`该操作超出 AI 边界，需用户手动执行` |
| Git 冲突 | 报告：`检测到 Git 冲突，请手动解决后重试` |
| 代码文件不存在 | 报告：`sources/xxx 不存在，请确认路径` |
| frontmatter 格式错误（现有笔记） | 报告并询问是否修复 |

## 7. 性能与批量操作

- **单次 SOP 创建的笔记数上限**：10 条（超出需分批）
- **Git 提交粒度**：每个 SOP 独立提交，不要把多个不相关的操作混在一个 commit
- **文件读取上限**：单次会话读取不超过 50 个文件（避免上下文溢出）
